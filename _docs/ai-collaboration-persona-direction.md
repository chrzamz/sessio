# AI Session Hub：AI 协作人格 / 工作画像方向说明

创建日期：2026-05-11
项目：`/Users/chrzam/Workspace/side_project/ai-session-hub`

## 1. 背景

AI Session Hub 当前已经完成了最核心的底座：

- 扫描本地 Claude Code / Codex JSONL 对话
- 写入 SQLite 索引
- 支持全文搜索
- 支持按工具、按项目过滤
- 支持查看对话详情

这意味着项目已经不仅仅能“找回历史”，还具备了进一步做“对话分析层”的条件。

本次探索的结论是：

这个产品后续最有意思、也最有差异化的方向，不只是继续做检索，而是增加一层 AI 分析能力，用来从长期对话中提炼用户的：

- 工作方式
- 决策风格
- 项目习惯
- AI 工具使用分工
- 长期优势 / 风险模式
- 近似的协作人格画像

注意：这里的重点不是做心理学诊断，而是做“工作行为画像”。

---

## 2. 这个方向为什么值得做

### 2.1 从“聊天记录管理器”升级成“认知工作台”

如果只做检索，AI Session Hub 的价值主要是：

- 找得到过去的会话
- 搜得到某个关键词
- 能回看某次讨论

这当然有用，但仍然偏工具层。

如果加入分析层，产品就能进一步回答更高价值的问题：

- 我最近 30 天到底在和 AI 一起做什么？
- 我更常把 AI 用于实现、调试、表达、还是判断？
- 哪些项目只是高活跃，哪些项目是真正深推进？
- 我最常见的工作模式是什么？
- 我最容易卡住、反复、或者拖延的地方是什么？

这会让产品从“记录管理工具”变成“用户与 AI 协作方式的观察仪表盘”。

### 2.2 这个方向天然适合当前项目

当前项目已经有：

- session 层数据
- message 层数据
- tool / project / timestamp 等元信息
- SQLite 查询能力

所以 AI 分析不是凭空增加功能，而是自然建立在现有索引层之上。

### 2.3 这个方向的差异化足够强

普通聊天产品也能做“摘要”。

但 AI Session Hub 更有价值的地方是：

- 跨工具（Claude Code / Codex）
- 跨项目
- 长周期
- 面向真实工作流

因此它可以做出的不是“单次总结”，而是“长期协作画像”。

---

## 3. 方向定义

一句话定义：

AI Session Hub 不只是管理 AI 对话，而是分析“用户如何与 AI 一起工作”。

更具体地说，这一方向聚焦于两个层面：

### 3.1 工作行为画像

从对话中提炼：

- 工作风格
- 项目推进方式
- 决策方式
- 工具使用偏好
- 结构化程度
- 多项目切换模式

### 3.2 协作人格画像

从长期对话习惯中，推断：

- 你更像哪类使用者
- 你的典型优势是什么
- 你的风险点是什么
- 你最容易掉进的工作模式是什么

注意：

这里的“人格”不是严格心理学人格，而是“与 AI 协作时表现出的稳定工作人格”。

---

## 4. 这类分析能产出什么

### 4.1 可以较高可信度产出的内容

这些属于“行为层”，可信度相对高：

- 更偏产品 / 技术 / 写作 / 调试 / 分析中的哪几类任务
- 更喜欢长协作还是短协作
- 更常用哪个工具做深任务，哪个工具做轻任务
- 更像单项目深耕还是多项目切换
- 更偏先结构化再执行，还是先执行再补结构
- 最近一段时间的主题重心变化

### 4.2 可以中等可信度推断的内容

这些属于“风格层”，需要明确是推测：

- 决策风格
- 推进风格
- 常见风险模式
- 拖延 / 分散 / 过度优化的倾向
- 高层次优势（如抽象能力、拆解能力、表达能力）

### 4.3 不应该装作很准的内容

这些不适合被包装成精确结论：

- 严格 MBTI 类型
- 心理学意义上的人格诊断
- 情绪健康判断
- 人生选择建议

产品里如果要出现 MBTI，只能作为“娱乐化、低置信度的近似映射”。

## 5. 我们已经验证到的有趣洞察样例

基于当前真实索引数据，已经能看出一些有产品感的东西：

### 5.1 工具角色分工

例如可以判断：

- Claude Code 更像深协作工具
- Codex 更像轻量试探工具

这类结论可以做成“AI 工具协作画像”。

### 5.2 项目热度与深度并不相同

例如可以区分：

- 高频但浅层的入口项目
- 低频但高浓度的深工项目

这类结论适合做成“项目推进强度图”。

### 5.3 用户可能的风险模式

例如可以识别：

- 容易多线程切换
- 容易在结构优化里延迟收尾
- 容易把 AI 的高质量反馈变成新的任务分支

这类结论很适合做成“风险提醒卡片”。

### 5.4 用户的优势画像

例如可以提炼：

- 擅长结构化模糊问题
- 能在产品 / 技术 / 表达之间切换
- 行动导向强
- 对高层方向与本质问题敏感

这类结论适合做成“协作人格报告首页”。

## 6. 产品边界：这不是心理测评产品

这个方向很有吸引力，但必须严格设边界。

建议明确以下原则：

1. 不做心理诊断
2. 不输出装作绝对准确的人格分类
3. 所有结论尽量基于可解释证据
4. 优先输出“行为观察”，其次才是“风格推断”
5. 用户应能看到结论背后的依据

建议的文案风格是：

- “从过去 90 天与你和 AI 的对话中，我们观察到……”
- “高概率倾向于……”
- “这更像一种工作模式，而不是固定人格”

避免写成：

- “你就是……”
- “你的 MBTI 一定是……”
- “你的性格本质上……”

## 7. 建议的功能分层

### Phase A：规则驱动画像（低成本，先验证）

目标：先不用 LLM，基于现有 SQLite 数据和关键词规则，做第一版画像。

可做功能：

- 工具使用画像
- 项目活跃 / 深度画像
- 长会话比例
- 多项目切换程度
- 写作 / 调试 / 产品 / 自动化等任务类别分布
- 长期 openers / 高频词 / 高频主题的基础统计

价值：

- 成本低
- 不引入模型依赖
- 先验证用户是否觉得“有趣”

### Phase B：会话级 AI insight 抽取

目标：对单个 session 提炼结构化 insight。

建议字段：

- summary
- task_type
- key_decisions
- blockers
- open_loops
- deliverables
- confidence

价值：

- 为更高层分析提供中间层数据
- 可被项目级和跨项目分析复用

### Phase C：跨项目人格 / 工作方式分析

目标：把多个 session insight 聚合成人的长期画像。

建议输出：

- 工作风格
- 决策方式
- AI 使用方式
- 优势 / 风险
- 近期重心迁移
- 低置信度协作人格标签

价值：

- 这是最有产品感的一层
- 也是最需要谨慎设计的一层

## 9. 页面 / 交互建议

### 9.1 Persona 页面

可以作为一个独立 tab，展示：

- 一句话总评
- 工作风格标签
- 最近 30 / 90 天变化
- 优势卡片
- 风险卡片
- AI 使用方式卡片
- 证据来源（可点击回看会话）

### 9.2 Explainability（可解释性）

每条结论都应该能展开看到：

- 依据了哪些统计
- 引用了哪些 session
- 是规则生成还是 AI 推断
- 置信度如何

### 9.3 语气

建议语气：

- 像一个冷静的工作分析师
- 不要像星座 / 心理测试小作文
- 不要过度煽情
- 要承认不确定性

## 10. 推荐的首个 MVP

如果现在开始做这一方向，建议第一版只做：

1. 工具协作画像
   - Claude Code / Codex 使用分布
   - 平均 session 深度
   - 长会话比例

2. 项目推进画像
   - 热度排行
   - 深度排行
   - 多项目切换程度

3. 任务类型画像
   - 产品 / 技术 / 写作 / 调试 / 自动化占比

4. 风险 / 优势提示（规则版）
   - 多线程切换倾向
   - 结构先行倾向
   - 深协作倾向
   - 高表达要求倾向

5. 一份简版 AI 协作人格报告
   - 工作风格
   - 决策风格
   - 工具分工
   - 优势 / 风险
   - 低置信度 MBTI 猜测（可选）

不要第一版就做：

- 实时监听
- 精细时间线人格演化
- 情绪分析
- 严肃心理学测评
- 太多炫技式图表

---

## 11. 为什么这个方向值得保留

这不是“顺手加个 AI feature”。

它更像是 AI Session Hub 的第二增长曲线：

第一阶段：帮用户管理和检索对话历史
第二阶段：帮用户理解自己如何与 AI 一起工作

后一层才真正有机会从工具变成产品。

一句话总结：

AI Session Hub 最终不只是“找回你说过什么”，
而是“帮你看清你和 AI 是怎么一起工作的”。

---

## 12. 下一步建议

建议后续按这个顺序推进：

### Step 1
先做不依赖 LLM 的规则版画像页，验证用户是否觉得有趣。

### Step 2
如果用户喜欢，再增加 `session_insights` 作为 AI 中间层。

### Step 3
最后再做跨项目聚合的人格 / 工作方式分析。

这样风险最小，也最符合当前项目“先最小实现、再迭代”的原则。
